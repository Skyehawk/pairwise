Introduction

This help page covers a range of topics around Singular Value Decomposition (SVD) and how to use your collected data with the pairwise SVD File Processing Tool.

The concept of SVD can be illustrate with the transitive property, which asserts that if a>b and b>c, then it follows that a>c. Now, in reality the linear algebra behind the scenes is a tad bit more complex to handle dozens of observations, but the general illistrative concept is the same. The Tool accepts a .csv file that contains the following information in the first three columns (No headers!):

The better team, the worse team, by how much (more on this in Getting Started).

Underpinning Concepts

Singular Value Decomposition (SVD) Example: Ranking UK Football Clubs

Step 1: Data Collection and Comparison Matrix

You have a list of pairwise comparisons of clubs that can be turnd into a matrix, but it might have missing values due to incomplete comparisons/observations.

  • Manchester United is better than Liverpool.
  • Liverpool is as good as Chelsea.
  • Chelsea is better than Arsenal.
  • Arsenal is as good as Manchester United.

Manchester United Liverpool Chelsea Arsenal
Manchester United - 1 -1 0
Liverpool -1 - 0 1
Chelsea 1 0 - -1
Arsenal 0 -1 1 -

Step 2: Singular Value Decomposition (SVD)

Just as a coach might analyze a match by breaking down different aspects like passing accuracy, defense, etc., SVD dissects the comparison matrix into three new matrices:

U Matrix (Players' Skills)

This matrix represents the style or characteristics of each club. Each row represents a club, and each column represents a particular "style" or "skill" that contributes to fans' preferences.

Factor 1 Factor 2 Factor 3
Manchester United 0.5 0.3 -0.7
Liverpool -0.6 0.4 0.2
Chelsea 0.6 0.6 0.3
Arsenal 0.1 -0.6 0.6

S Matrix (Skill Level Importance - Diagonal Matrix)

This is a diagonal matrix showing the importance or weight of each skill/style in influencing fan preferences. It highlights which aspects are more crucial in determining the rankings.

2.5 0 0 0
0 2.1 0 0
0 0 1.8 0
0 0 0 0.9

Vt Matrix (Game Strategies)

Similar to different strategies in a soccer match, this matrix represents how clubs employ these skills or characteristics. Each column shows how much a club relies on each "style" or "skill."

Factor 1 Factor 2 Factor 3
Manchester United 0.5 -0.7 -0.4
Liverpool -0.5 -0.6 0.6
Chelsea 0.6 0.4 -0.6
Arsenal 0.3 0.1 0.9

Step 3: Completing the Matrix

Just as a coach might predict a player's performance in a match based on their overall skills, SVD helps fill in missing comparisons by estimating how clubs would fare against each other based on their identified "styles."

Step 4: Final Rankings (Based on Completed Matrix)

After completing the matrix and calculating rankings, the inferred rankings are:

  1. Chelsea: 0.6
  2. Manchester United: 0.2
  3. Liverpool: 0
  4. Arsenal: -0.2

Once the matrix is completed, SVD provides a nuanced view of clubs' preferences beyond direct comparisons. It's like understanding how different playing styles or strategies contribute to fans' overall preferences, giving a holistic view of why one club might be favored over another.

These rankings indicate the clubs' standings based on the inferred preferences among fans, with Chelsea being the most preferred and Arsenal being the least preferred in this hypothetical scenario.

Features

Pairwise offers support for weighting the observations. Instead of specifying a > or = resulting in something like:

"Team1" > "Team2" = "Team3"

Which is the same as:

  • "Team 1" is better than "Team 2" by 1
  • "Team 2" is better than "Team 3" by 0
  • "Team 1" is better than "Team 3" by 1
We can place weights on those observations, resulting in:
  • "Team 1" is better than "Team 2" by 3
  • "Team 2" is better than "Team 3" by 0
  • "Team 1" is better than "Team 3" by 3

Singular Value Decomposition (SVD) with Weights Example: NHL Hockey Teams Ranking

1. Observations & Weighting:

The observations indicate the relative strengths between NHL teams, with specified units signifying the magnitude of preference between pairs. Let us suppose that over the course of a month the following observastions took place.

  • MTL is better than TOR by 3 units
  • BOS is better than NYR by 2 units
  • MTL is better than BOS by 1 unit
  • TOR is better than NYR by 3 units
  • MTL is better than NYR by 2 units
  • BOS is better than TOR by 1 unit
  • TOR is better than MTL by 2 units
  • NYR is better than BOS by 1 unit
  • MTL is better than TOR by 1 unit
  • BOS is better than NYR by 3 units

2. Matrix Construction:

The weighted observations translate into a comparison matrix reflecting the weighted differences between teams, averaging applied where a pair of teams had multiple observations (although you could look at just the last observation or make more recent observations weight heavier in the average):

MTL TOR BOS NYR
MTL 0 2 0.5 1.5
TOR -2.5 0 -0.5 2.5
BOS -0.5 0.5 0 -2
NYR -1.5 -2.5 2 0

3. Singular Value Decomposition (SVD):

SVD considers these weighted observations to identify underlying relationships between teams based on their weighted comparisons. The units impact the construction of the matrices (U, S, Vt) in the SVD.

4. Impact of Units:

Larger units in the comparison matrix suggest stronger preferences between teams, potentially influencing the resulting matrices (U, S, Vt) in the SVD process and the inferred rankings among the teams.

  1. TOR (Toronto Maple Leafs): 1.125
  2. BOS (Boston Bruins): 0.75
  3. MTL (Montreal Canadiens): 0.375
  4. NYR (New York Rangers): 0.0

5. Ranking Inference:

The rankings derived from the completed matrix consider the weighted observations to determine the relative s trengths of the NHL teams. Teams with larger weighted differences (higher units) in their favor might reflect higher ranks compared to those with smaller differences.

FAQs & Troubleshooting

Direct users on where to ask questions

FAQs

FAQ 1: Do you store the data submitted?
Answer 1: By default no, no data or results are ever saved server side without the user's consent. If you chose to submit the data anonymously for improvment of the tool a copy is saved along with the optional memo (i.e. you could include team #, event code, or what the data represent such as endgame strength)

FAQ 2: Can I run this tool offline?
Answer 2:: Yes! The source code for the bare algorithm is avaliable as a Google Colab notebook

FAQ 3: My team already has plenty of scouts and a data collection process, what can Pairwise do for me?
Answer 3: For teams with robust data collection already in place, it makes sense for Pairwise not to generate any sort of picklist that would go on the field for selection, but rather serve a supplementary role. Teams could have scouts for Red and Blue alliances submit a ranking of the three robots on those alliances; this encourages some team analysis while the match is still fresh. Then, Pairwise can generate a picklist to expedite scouting meetings so everyone can get to bed. Pairwise may also have applications where there is not time for long scouting meetings (MI District Champs), single-day events, or offseason events with no published schedule.

Troubleshooting

Issue: The tool generated a completely empty results file, what happened? Where are the rankings?
Cause: This is the result of a set of observations that could not converge in SVD.
Things to try: If you have observations in this form:

"Team 1" is better than "Team 2" by 3

Try changing to binary observations, i.e.

"Team 1" is better than "Team 2" by 1 or "Team 1" is better than "Team 2" by 0

This likely will provide more bandwidth for the underlying matrix to converge. A preprocessing tool may be developed at some point to aid users with data that does not converge, but does not currently exist.

Additional Resources

Pairwise operates on the principle of Rank Recovery from Limited Pairwise Observations using Low Rank Matrix Completion which is implemented through Singular Value Decomposition with the aid of Scipy v1.11 and Python v3.10. Operationally, larger teams with enough members to scout in FRC may find use in implementing Pairwise into the scouting workflow similar to how FRC 1678 (Citrus Circuits) has in the past.
